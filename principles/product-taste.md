# Product Taste Principles

## Product Taste Is Judgment

Good product taste means deciding what should exist, what should be hidden, and what should be refused.

Ask before every feature:
- What user job does this serve?
- What current workflow does it replace or improve?
- What decision becomes easier?
- What can be removed because this exists?
- What observable behavior proves it worked?

## Use Jobs To Be Done

```text
When [situation],
I want to [make progress],
so I can [outcome].
```

Do not define the job as using the feature. Define it as user progress outside the app.

## Prefer Workflows Over Feature Lists

Default product flow:

```text
Input
  -> Diagnosis
  -> Decision
  -> Execution
  -> Evidence
  -> Feedback
  -> Next iteration
```

Features outside the workflow need stronger justification.

## Behavioral Metrics

Prefer:
- Core workflow completion
- Time to usable output
- Copy/export rate
- Error recovery rate
- Repeat use
- Feedback submission
- Retention or follow-up action

Avoid using visual polish or feature count as the main success signal.
